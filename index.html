Operation System Project Report: Robot bunbo

I. Description of the architecture of the robot

The robot uses 3 sensors which are: color, ultra sonic, and gyro.

  • The color sensor is used for reading black line on the ground where the robot go.
  • The ultra sonic sensor is used to calculate the distance.
  • The gyro sensor is used to calibrate the robotâ€™s direction at any moment.

The robot uses 3 motors to move and to grab the ball

  • The 2 main motors are used for 2 wheels, the 2 wheels move independently so that the robot can turn.
  • The grabber motor is situated between the 2 main motors. The grabber motor acts as a trap door to trap the ball inside the robot.

II. Algorithms used for the two most important points

In order to make the robot run straight we continuosly calibrate the speed of each wheel with respect to the gyro sensor. If the robot goes to the left, we decrease the speed of the left a little and vice versa to ensure that the robot goes straight when it is supposed to.

In order to make the robot turn in the right direction, we use the gyro sensor to calculate exactly the turn angle. We also save the true straight angle so that we can go back to the basic angle whenever we want to.

We structure our program to be a loop that will sequentially call update sensor value and drive the robot. This architecture, which is usually seen in game programming, proves to be very effective as the sensor information is always up to date and each stages of the program is controllable and modularize for easy of debugging and maintenance.

We structure the game into steps that can be programmed independently. Each step correspond to each stage of the competition. For example the beginner of the small stadium can go until the black line, turn left, scan for the ball â€¦ Each step is programmed separately so that we can ensure the success of each step then we integrating them into a series of steps that can play the game.

We use a structure that is similar to that of object oriented programming and the command design pattern. This structure allows us to define fine grain steps that can be reuse in both the big stadium and the small stadium without re-programming too specificly. We design an interface struct called â€œstepâ€ that contain function pointers: run_motor, update, initialize. The initialize function is called at the beginning of the step the update and run_motor function is called sequentially in loop during the step if the step finish the handle is pass to the next step.
